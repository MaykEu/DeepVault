# DeepVault — AI Contributor Reference

Build and maintain the DeepVault web application. Pure vanilla HTML/CSS/JS — zero dependencies, runs by double-clicking `index.html`, deployed via GitHub Pages.

## Content Quality Standards — Pedagogical Philosophy

DeepVault is a **computing education platform** — programming languages, computer systems, game engines, networking, cybersecurity, algorithms, GPU programming. The same content quality principles apply regardless of which computing domain you're teaching.

### What Makes a DeepVault Note

Lessons, not summaries. Every note must follow these rules:

1. **Two-Way Explanation (MANDATORY):** Simple analogy section + Technical deep-dive section. Both REQUIRED.
2. **No shallow summaries:** Deep coverage — 700-1100 line notes are normal.
3. **Hardware-first / Concrete-first:** Start from physical reality before abstractions.
4. **Prerequisites callout (MANDATORY):** List prerequisite notes with wiki links.
5. **See Also (MANDATORY):** Minimum 3 wiki-linked related notes at the end.
6. **Exercises:** 4-5 per note: conceptual, code/application, analysis, real-world.
7. **Show the trap:** Demonstrate WRONG patterns alongside correct ones.
8. **Analogies → Math → Code:** Physical metaphor first, then notation, then implementation.

### What DeepVault Is NOT

- ❌ Reference documentation — belongs in a wiki
- ❌ Cheat sheets or bullet-point summaries
- ❌ Lecture transcripts
- ❌ Beginner tutorials that skip the "why"

### Why This Vault Exists

Most computing education is either too shallow or too jargon-dense. DeepVault bridges both — every concept explained twice, grounded in hardware reality, connected via wiki links. The pedagogy IS the product.

## Architecture

**SCALABLE BUILD SYSTEM.** `js/data.js` is generated by `python build.py` from 4 independent JSON source files in `data/`:

- `data/notes.json` — all note content (vault → build.py). **Never edit manually.**
- `data/quizzes.json` — all QUIZ_DATA + QUIZ_NOTES. **Write here safely — can't corrupt notes.**
- `data/folders.json` — folder structure per domain.
- `data/reference.json` — glossary, guide, learning paths.

Build process: `python build.py` validates all quizzes → generates `js/data.js`. **If validation fails, build ABORTS.**

```
data/
├── notes.json      ← Note content (vault → build.py)
├── quizzes.json    ← Quiz questions (safe to edit)
├── folders.json    ← Folder structure
├── projects.json   ← Project notes (vault → convert.py)
└── reference.json  ← Glossary, Guide, Learning Paths

build.py            ← Combines sources → js/data.js (validates first)
quiz-cli.py         ← Safe quiz management (add, verify, backup)
test.js             ← Smoke tests (syntax, CSS, quiz format)
convert.py          ← Vault → data/*.json converter
js/data.js          ← GENERATED — never edit manually
```

**VAULT FIRST — ALWAYS.** Every content change MUST happen in the vault `.md` file FIRST, then sync to data.js. The vault is bundled in this repo at `Obsidian/Game Development/` — this is the canonical source of truth for all note content, quizzes, and learning paths. Never create or edit content directly in `data/notes.json` — always write the `.md` file in the vault first, then rebuild.

**PROJECTS PIPELINE.** Notes inside `*/Projects/` folders in the vault are automatically detected by `convert.py`, written to `data/projects.json`, and surfaced in the 🔨 Projects tab via `PROJECT_NOTES` in `data.js`. No quizzes needed — projects are build-work, not test-work. To add a project: create `Category Name/Projects/My Project.md` in the vault, run `python convert.py`. Everything else is automatic.

## Learning Path Structure

All Learning Paths follow a consistent format:

- **Main Path** — the core sequence of numbered notes. Follow in order. Contains only the essential notes for each topic.
- **Deep Dives** — optional depth. Each deep dive is a clearly labeled section saying "Take after **NN — Note Name**". These are supplementary — skip them and continue the main path, or take them for deeper understanding.
- **Chapter Summaries** — synthesis notes that connect the dots after completing a section.
- **`—` notation** — used ONLY for overview notes (labeled "Overview — read first") and reference notes (labeled "Reference"). Never used for deep dives or branches.

**PITFALL:** Do NOT interleave deep-dive notes into the Main Path table with `—` numbers. Every deep dive belongs in its own clearly separated section. The `—` symbol must never be ambiguous — it always means either an overview or a reference note.

## Quiz Quality Rules

- **5-15 questions per note MANDATORY.** 1-2 question quizzes are UNACCEPTABLE. Mix MC and text input.
- Every question MUST have an explanation
- FIELD NAMES MUST BE FULL: `question` NOT `q`, `options` NOT `o`, `correctAnswer` NOT `a`, `explanation` NOT `e`. Abbreviated fields cause "undefined" — this broke 115 quizzes.
- JSON ESCAPING: Do NOT backslash-escape single quotes (`\'`). Use `json.dumps()` for ALL JSON output.
- **MC LENGTH BIAS: correct answer must not be the uniquely longest option.** Users can guess by picking the longest answer. Target <5%. Fix: pad shorter wrong options with comparable detail, make 1-2 wrong options slightly longer, or use trailing whitespace (collapsed in HTML rendering, invisible to users). **PITFALL: never `.rstrip()` quiz options after fixing — stripping undoes the bias elimination (observed: 0% → 78%).**
- WRITE to `data/quizzes.json`, then run `python build.py`. Never leave intermediate batch files on disk.
- VERIFY: Run `python quiz-cli.py verify` → 0 errors AND 0 warnings. Run `node test.js` → 20/20.

## Quiz Creator Checklist

When instructing an AI to create quizzes, these MUST be in the prompt:

1. **FIELD NAMES:** Use FULL names — `question`, `options`, `correctAnswer`, `explanation`. NEVER abbreviate to `q`/`o`/`a`/`e`.
2. **JSON ESCAPING:** Use `json.dumps()`. Do NOT backslash-escape single quotes.
3. **BUILD AFTER WRITING:** Write to `data/quizzes.json`, run `python build.py` to regenerate `js/data.js`.
4. **UPDATE QUIZ_NOTES:** After adding to the `quizzes` dict, MUST rebuild the `quiz_notes` dict in `data/quizzes.json` — it maps `folder_id → [note names with quizzes]`. Without this, quizzes exist in data but won't appear in the Study tab. Cross-reference `data/notes.json` folder assignments with quiz names.
5. **VERIFY:** After every batch, run `python quiz-cli.py verify`. Fix errors before next batch.
6. **NON-STOP INSTRUCTIONS:** AI assistants naturally pause between batches. Include: "DO NOT PAUSE or ask to continue. Process ALL notes until complete."

### Quiz Notes Routing — Two Fields Must Stay In Sync (CRITICAL)

`data/quizzes.json` has TWO independent fields:

```json
{
  "quizzes": { "Note Name": {"questions": [...]} },
  "quiz_notes": { "computer-systems": ["Note A", "Note B", ...] }
}
```

- **`quizzes`** → `QUIZ_DATA` (actual questions). `build.py` reads this to generate `QUIZ_DATA['Note Name'] = {...}`.
- **`quiz_notes`** → folder → list of note names with quizzes. `build.py` reads this to generate `QUIZ_NOTES[folder] = [...]`. The site uses `QUIZ_NOTES` to decide which notes go to the **Study tab** vs **Guides tab**.

**PITFALL:** Adding quizzes only to the `quizzes` field leaves `quiz_notes` stale. The build passes, `quiz-cli.py verify` passes, but the Study tab shows old counts — notes with quizzes still appear under Guides. **This is invisible to automated verification** unless you specifically check `quiz_notes` counts per folder.

**Symptom:** User says "I don't see the new quizzes in the Study tab." Quiz exists everywhere but `QUIZ_NOTES[folder]` doesn't list the note.

**Fix:** Rebuild `quiz_notes` from the `quizzes` dict by cross-referencing note names with their folder assignments in `data/notes.json`:

```python
from collections import defaultdict
quiz_notes_new = defaultdict(list)
for name, note in notes.items():
    if name in quiz_names:
        quiz_notes_new[note.get('folder', 'computer-systems')].append(name)
quiz_data['quiz_notes'] = dict(quiz_notes_new)
```

### Overview Notes Policy

**Overview notes (00 — prefix or "Overview" in title) have NO quizzes.** They appear in Learning Paths as navigation aids but are excluded from Study and Guides tabs. This keeps the Study tab focused on testable knowledge.

### Quiz Engine Configuration

The quiz engine uses `questionsPerQuiz: 15` to draw N questions from a shuffled pool (min of N vs pool size). Previously hardcoded at 8. The `findCorrectIndex()` helper normalizes answer matching when `correctAnswer` isn't an exact string match in `options` (e.g., trailing whitespace differences).

## Table Rendering — Empty First Column Bug

**Tables with an empty first header cell render with headers shifted left.** Fix: give the first column a label (`| **Property** | **FName** |`). Do NOT use `&nbsp;` — renders as literal text.

## Angle Bracket Bug — Web Renderer (HTML Parser)

**Different from the Obsidian angle bracket bug.** Inside the web renderer, `<text>` inside backtick code spans gets interpreted as an HTML tag by the browser and STRIPPED from rendered output.

**Example:** `` `class Derived : public Base<Derived>` `` renders as `class Derived : public Base` — the `<Derived>` is eaten by the HTML parser.

**Fix:** Use `&lt;` and `&gt;` in source markdown: `class Derived : public Base&lt;Derived&gt;`. The browser renders these correctly inside `<code>` tags.

## Submit Button & Review Mode

Two edge cases: submit button silently fails (correctAnswer vs correctIndex mismatch) and review mode shows empty state when questions exist (q vs question field mismatch). Both require dual-format support. See `references/submit-review-pitfalls.md` and `references/quiz-format-pitfalls.md`.

## Hash Routing

Same-hash navigation bug: `router.navigate` to current hash is ignored. Fix: append `?t=timestamp` to quiz routes. Also: NEVER build onclick strings via Python — multi-level escaping nightmare. Use router-based solutions. See `references/hash-routing-pitfalls.md`.

## Scroll Tracking — Per-Note Position Persistence (2026-07-09)

The `LearnView.trackScroll` and `LearnView.restoreScroll` methods save and restore scroll position per note.

**Save triggers (4 layers):**
1. **Scroll debounce (300ms)** — saves during reading
2. **`router.navigate` wrapper** — saves immediately BEFORE any SPA link click (before hash changes, while correct scrollY is still available)
3. **`hashchange` capture listener** — clears `_tk` to prevent dashboard overwrites (see below)
4. **`beforeunload`** — catches tab close / refresh

**Restore:** Uses double `requestAnimationFrame` (not `setTimeout`) to ensure DOM settles after `innerHTML` replacement before `window.scrollTo(0, position)`.

**CRITICAL — `_tk` clearing:** After navigating away from a note, `_tk` is set to `null`. Without this, `_tk` retains the last note's key. When the user clicks "Learn" from the dashboard, the wrapper saves the dashboard's `scrollY` (0) under the note's key — overwriting the real saved position. `restoreScroll` then reads 0 and scrolls to top.

**Architectural insight:** `hashchange` fires too late in bubble phase (router listener runs first). The CAPTURE phase listener (`true` flag) fires BEFORE the router, saving position while old content is still visible. The save was moved OUT of the capture handler because the browser scrolls to top when the hash changes to `#/` — the capture handler was overwriting the correct position with 0.

## `href="#"` Overrides SPA Hash Navigation — CRITICAL PITFALL (2026-07-09)

**`<a href="#">` will silently break your SPA navigation.** After the onclick handler calls `router.navigate('#/some/path')` and sets the hash, the browser follows the `href="#"` target, setting `window.location.hash` to `#` (empty). The router sees an empty hash and renders the dashboard. **No errors, no console warnings.**

**Fix:** Use `href="javascript:void(0)"` on ALL navigation links. Wiki-links and footnotes already have `event.preventDefault()` — their `href="#"` is harmless.

**Files fixed:** `learn.js` (3 back-links), `topic-hub.js` (1), `quiz-engine.js` (1), `summary.js` (1).

## convert.py — Common Bugs & Fixes

`convert.py` reads vault `.md` files and writes `data/*.json` source files. Key pitfalls:

- **VAULT path:** The `VAULT` constant must point to the correct vault location inside the repo (`Obsidian/Game Development/`).
- **Quiz extraction:** Old regex `r"QUIZ_DATA\['([^']+)'\]\s*=\s*(\{[\s\S]*?\n\})"` fails on nested `{}`. Replaced with brace-counting loop that scans until `};` at depth 0.
- **List aliasing (2026-07-06):** `flat = group.get('', [])` creates an ALIAS, not a copy. `flat.extend()` then mutates the original in-place, re-adding subfolder notes after post-processing. Fix: `flat = list(group.get('', []))`.
- **Variable typos:** Always run `python convert.py` after patches — it catches these immediately.

## Wiki-Link Alias Display

LP tables showing raw `path|alias` text instead of clean note titles. Fix: extract `displayName` in `md2html`'s wiki-link callback, use across all return paths. See `references/wiki-link-alias-display.md`.

## Text Input — Strip Spaces Before Comparing

Binary/hex answers commonly include spaces (`1111 1111` vs `11111111`). Strip `/\s/g` from user answer AND correct answer before comparing. Also strip from `acceptableAnswers[]`. Without this, correct spaced answers are marked wrong.

## Nested Groups — Parent/Child Rendering

Groups with `/` in their name (e.g., `Physics/Collision Detection`) render as nested parents wrapping child groups. When `FOLDER_GROUPS` has both flat and slash-prefixed keys with the same parent, the renderer merges them into ONE section. See `references/nested-groups.md`.

## Study Tab Numbering — Per-Group Reset

Numbering resets to 01 for each group. Guides tab notes never get numbered.

## Reference Content — Guide & Glossary

The Guide lives in `guide-content.txt` (plain markdown). `convert.py` loads it and writes into `reference.json`. The Glossary lives in `data/reference.json` (sourced from the vault's `Glossary.md`). Both render via `LearnView.renderRef()`.

**When editing:** edit the source file (`guide-content.txt` or vault `Glossary.md`), run `python convert.py`, verify in browser.

## Adding a New Category

1. Create a folder in the vault with `.md` notes
2. Add one entry to `FOLDER_MAP` in `convert.py`:
   ```python
   ('category-id', 'Display Name', 'Vault Folder', '🩺', '#ff6b6b'),
   ```
   Format: `(id, display_name, vault_folder, icon, color)`
3. Run `python convert.py` — everything else is automatic

## Adding a Note

1. Write the `.md` file in the vault first — create it there, not in `data/notes.json`
2. The note's **filename** (without `.md`) becomes the key in `NOTES_CONTENT`
3. Add to `data/notes.json` and `data/folders.json`
4. Run `python build.py` to regenerate `js/data.js`
5. Verify: refresh `index.html`, navigate to the note — no "(coming soon)"

## Working With AI Assistants

This entire project was built with AI collaboration. The build system is designed for AI safety: `data/quizzes.json` is the only file AI contributors should write to directly — it can't corrupt note content. Everything else flows through `build.py` validation.

When using an AI to contribute:

1. **Load this SKILL.md first** — it prevents the exact bugs from dozens of commits of development
2. **Give explicit non-stop instructions** — AI assistants naturally pause between batches
3. **Verify after every batch** — run `python quiz-cli.py verify` and `python build.py`
4. **Never skip verification** — silent failures (like stale `quiz_notes`) are invisible until users report them
